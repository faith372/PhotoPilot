# PhotoPilot AI 项目开发文档

版本：v0.1  
日期：2026-06-01  
项目定位：AI 照片筛选与修图助手

## 1. 项目概述

PhotoPilot AI 是一个面向个人创作者、摄影师、电商运营和普通照片用户的 AI 照片工作台。项目目标是帮助用户从大量照片中快速挑出好片，并根据自然语言意图完成基础调色、美颜和风格化处理，降低照片整理与后期修图门槛。

项目当前形态为浏览器静态原型，已覆盖照片导入、示例照片、AI 模拟评分、筛选、基础调色、美颜参数、自然语言修图方案生成和方案导出等核心体验。后续版本将接入真实 AI 能力、图像处理引擎、批量导出、桌面端封装和隐私模式。

## 2. 产品目标

### 2.1 核心目标

- 让用户在导入大量照片后，能快速看到 AI 推荐的保留、待定、淘汰结果。
- 让 AI 给出可解释的选片理由，包括清晰度、曝光、构图、色彩、表情、重复照片等维度。
- 让用户用一句话表达修图意图，例如“日系清透一点，皮肤自然，不要太假”，系统自动转换为可编辑参数。
- 支持基础调色、美颜、风格化预设和非破坏式编辑。
- 保持低门槛体验，打开后第一屏就是可用的照片工作台。

### 2.2 非目标

- MVP 阶段不做专业级 RAW 调色软件的全量功能。
- MVP 阶段不做复杂图层、蒙版、精细液化和商业级人像重塑。
- 原型阶段不直接上传用户照片到云端，真实 API 接入前需要明确隐私提示与授权。

## 3. 目标用户与使用场景

### 3.1 目标用户

| 用户类型 | 主要需求 | 典型痛点 |
| --- | --- | --- |
| 普通用户 | 从旅行、聚会、自拍中快速挑图和轻修 | 照片太多，不会调色，怕修得不自然 |
| 内容创作者 | 快速选出适合发布的照片并统一风格 | 时间有限，需要稳定审美和批量效率 |
| 摄影师 | 对连拍、活动、客片进行初筛 | 重复照片多，人工初筛成本高 |
| 电商运营 | 快速处理商品图并统一交付效果 | 需要干净、统一、可批量复制参数 |

### 3.2 核心场景

1. 用户导入一组本地照片。
2. 系统自动分析照片质量并打分。
3. 用户按“保留、待定、淘汰”查看照片。
4. 用户选中某张照片，查看 AI 评分理由和修改建议。
5. 用户选择预设，或输入自然语言修图意图。
6. 系统生成调色与美颜参数，用户可手动微调。
7. 用户导出当前方案，后续版本可导出真实处理后的图片。

## 5. 用户流程

### 5.1 选片流程

1. 进入应用后默认展示照片工作台。
2. 用户点击“导入照片”选择本地图片，或点击“载入示例”体验流程。
3. 系统生成照片列表，并为每张照片计算 AI 分数。
4. 系统根据分数将照片划分为：
   - 保留：适合作为成片或优先精修。
   - 待定：有可用价值，但需要与同组照片对比。
   - 淘汰：清晰度、曝光、构图、表情或重复度问题明显。
5. 用户可调整最低分数、人像筛选、已编辑筛选和排序规则。
6. 用户可手动修改单张照片状态，覆盖 AI 推荐。

### 5.2 修图流程

1. 用户选中照片。
2. 右侧面板展示 AI 分数、评分理由和建议。
3. 用户选择自动调色预设，或输入自然语言修图需求。
4. 系统生成编辑参数：
   - 曝光
   - 对比
   - 色温
   - 饱和
   - 美颜
5. 用户在预览区切换原图和效果。
6. 用户可继续手动微调参数。
7. 用户导出当前编辑方案，后续版本导出处理后的图片文件。

## 6. 功能需求

### 6.1 照片导入

- 支持一次导入多张图片。
- 支持 JPG、PNG、WebP 等浏览器可读取格式。
- 导入后立即生成缩略图和预览。
- 本地原图不应被覆盖。
- 真实桌面端版本需要支持文件夹导入、拖拽导入和增量扫描。

### 6.2 AI 评分

AI 评分应从 0 到 100 输出综合分，并返回结构化维度：

| 维度 | 说明 |
| --- | --- |
| sharpness | 主体清晰度、虚焦、运动模糊 |
| exposure | 曝光是否稳定，高光和阴影是否可恢复 |
| composition | 构图是否平衡，主体是否明确 |
| color | 色彩是否干净，是否存在明显偏色 |
| expression | 人像表情、闭眼、眼神、姿态 |
| duplicate | 与同组照片是否高度重复 |

推荐规则：

- 82 分及以上：默认保留。
- 66 到 81 分：默认待定。
- 65 分及以下：默认淘汰。

### 6.3 AI 建议

AI 建议必须使用用户能理解的自然语言，不暴露复杂模型术语。建议内容应包括：

- 为什么推荐保留或淘汰。
- 有哪些明显问题。
- 是否适合继续修图。
- 推荐使用哪类调色或美颜方案。

示例：

```text
主体清晰，曝光稳定，适合作为精选。
肤色略暗，建议提高曝光并轻微降低对比。
疑似连拍重复，建议与相邻照片对比后保留一张。
```

### 6.4 基础调色

MVP 编辑参数：

| 参数 | 范围 | 默认值 | 说明 |
| --- | --- | --- | --- |
| exposure | -100 到 100 | 0 | 控制整体亮度 |
| contrast | -100 到 100 | 0 | 控制明暗反差 |
| temperature | -100 到 100 | 0 | 控制冷暖倾向 |
| saturation | -100 到 100 | 0 | 控制色彩浓度 |
| beauty | 0 到 100 | 12 | 控制自然美颜强度 |

编辑应采用非破坏式方案，即保存参数而不是覆盖原图。

### 6.5 风格化预设

首批预设：

| 预设 | 用途 | 参数倾向 |
| --- | --- | --- |
| 清透自然 | 人像、生活照、社交发布 | 提亮、低对比、轻美颜、略冷 |
| 胶片暖调 | 旅行、人文、婚礼纪实 | 暖色、中对比、低饱和 |
| 干净棚拍 | 商品、证件感人像、统一交付 | 主体提亮、低杂色、低饱和 |
| 夜景增强 | 夜景、人像霓虹、城市照片 | 高对比、冷色、增强色彩 |

### 6.6 自然语言修图

用户输入自然语言后，系统需要输出编辑方案。MVP 阶段可使用关键词映射，生产版本应接入 AI 生成结构化方案。

输入示例：

```text
日系清透一点，皮肤自然一点，背景冷一点，不要太假。
```

输出示例：

```json
{
  "exposure": 18,
  "contrast": -8,
  "temperature": -18,
  "saturation": 8,
  "beauty": 16,
  "notes": [
    "提高亮度并降低硬对比，让画面更轻。",
    "美颜强度保持克制，优先保留皮肤纹理。",
    "背景略微转冷，增强清透感。"
  ]
}
```

### 6.7 导出

原型阶段：

- 导出当前照片的编辑参数摘要。
- 明确提示不会覆盖原图。

生产阶段：

- 支持导出 JPG/PNG。
- 支持批量导出保留照片。
- 支持保留 EXIF 的可选开关。
- 支持导出命名规则和目标文件夹设置。
- 支持导出调色参数 JSON，便于复用和回滚。

### 7.3 AI Provider 接口设计

UI 不应直接依赖某个具体 AI 厂商，应通过 Provider 抽象层调用。

```ts
interface PhotoInput {
  id: string;
  fileName: string;
  mimeType: string;
  width?: number;
  height?: number;
  localPath?: string;
  imageData?: Blob;
}

interface PhotoScore {
  finalScore: number;
  status: "keep" | "maybe" | "reject";
  metrics: {
    sharpness: number;
    exposure: number;
    composition: number;
    color: number;
    expression?: number;
    duplicate?: number;
  };
  reasons: string[];
}

interface VisionScoringProvider {
  scorePhoto(input: PhotoInput): Promise<PhotoScore>;
  scoreBatch(inputs: PhotoInput[]): Promise<PhotoScore[]>;
}

interface RetouchPrompt {
  photoId: string;
  userPrompt: string;
  currentEdit: EditRecipe;
  photoContext?: PhotoScore;
}

interface EditRecipe {
  exposure: number;
  contrast: number;
  temperature: number;
  saturation: number;
  beauty: number;
  style?: string;
  notes?: string[];
}

interface RetouchPlanProvider {
  createPlan(input: RetouchPrompt): Promise<EditRecipe>;
}
```

## 8. 隐私与安全

照片属于高隐私数据，产品设计必须默认保护用户数据。

### 8.1 原则

- 默认本地处理。
- 上传云端 AI 前必须明确告知用户。
- 支持用户关闭云端 AI。
- API Key 不应写入前端代码或普通明文文件。
- 导出文件不得覆盖原图，除非用户明确选择。

### 8.2 生产版本要求

- 桌面端使用系统安全存储保存 API Key。
- 对上传任务显示文件数量、目的和服务提供方。
- 提供“仅本地模式”。
- 支持清除缓存、缩略图和历史项目。
- 错误日志不得包含原图、完整本地路径和 API Key。

## 9. 性能要求

| 场景 | 指标 |
| --- | --- |
| 导入 100 张普通 JPG | 缩略图生成和列表展示应在可接受时间内逐步完成 |
| 图片列表滚动 | 不应明显卡顿，后续需使用虚拟列表 |
| AI 批量评分 | 支持队列和进度展示，可暂停或重试 |
| 编辑预览 | 单张照片参数调整应接近实时反馈 |
| 批量导出 | 显示进度、失败原因和输出目录 |

## 10. 交互与视觉原则

- 第一屏必须是实际工作台，不做营销式落地页。
- 主要操作路径保持清晰：导入、选片、修图、导出。
- AI 结果必须可解释，减少黑箱感。
- 专业参数保留，但不压过默认的一键体验。
- 界面应适合长时间筛图，避免过强装饰和低信息密度。
- 每个关键操作都要有可恢复空间，例如重置编辑、手动修改状态。

## 11. 测试计划

### 11.1 功能测试

- 导入单张、多张、不同格式图片。
- 载入示例照片后能正常筛选、排序和预览。
- AI 重新评分后状态和统计数据同步更新。
- 手动修改保留、待定、淘汰后计数正确。
- 调整滑杆后预览效果、已编辑状态和统计数据同步更新。
- 原图/效果切换正确。
- 自然语言修图为空时不生成方案。
- 导出方案时包含当前照片名和编辑参数。

### 11.2 兼容测试

- Chrome、Edge、Safari、Firefox 基础可用。
- 桌面宽屏、平板尺寸、手机尺寸布局不重叠。
- 中文文件名能正常展示。
- 大尺寸图片不会导致页面崩溃。

### 11.3 安全测试

- 文件名中的特殊字符不会造成 HTML 注入。
- 本地服务器不能访问项目根目录之外的文件。
- API Key 不会出现在控制台日志和导出内容中。

## 13. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| AI 评分不稳定 | 用户不信任推荐结果 | 提供评分理由、允许手动覆盖、持续收集反馈 |
| 云端隐私担忧 | 用户不愿导入真实照片 | 默认本地模式、上传前确认、提供本地模型选项 |
| 大批量图片性能差 | 工作流卡顿 | 缩略图缓存、任务队列、虚拟列表、后台处理 |
| 修图效果不真实 | 用户认为 AI 过度处理 | 默认克制参数，保留手动调节和原图对比 |
| 多 AI 厂商切换成本高 | 技术绑定 | 从第一版真实接入开始使用 Provider 抽象 |

## 14. 验收标准

MVP 验收：

- 用户可以导入或加载至少 6 张照片并看到照片列表。
- 每张照片都有 AI 分数、推荐状态和理由。
- 用户可以按状态、最低分、人像、已编辑进行筛选。
- 用户可以选择照片并查看大图预览。
- 用户可以应用预设、拖动参数、输入自然语言生成方案。
- 用户可以在原图和效果之间切换。
- 用户可以导出当前编辑方案摘要。
- 页面在主流桌面浏览器中无明显布局错乱。

生产版第一阶段验收：

- AI 评分来自真实视觉能力，不再只依赖模拟规则。
- 编辑参数可以真实渲染并导出 JPG/PNG。
- API Key 通过设置页配置，不硬编码。
- 用户能明确知道照片是否会上传云端。
- 至少支持一个完整项目的保存、重新打开和继续编辑。

7. 评估 Electron 与 Tauri，选择桌面端封装路线。
